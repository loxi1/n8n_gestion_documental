🧠 1. ¿Cada cuánto revisa el Email Trigger (IMAP)?
El nodo Email Trigger en n8n NO usa cron como tal, trabaja en modo:
👉 “escucha continua” (IDLE)
Es decir:
Se queda conectado al servidor IMAP
Espera nuevos correos en tiempo real
Por eso se queda “pensando infinito” → y está bien ✔️
🚨 Entonces… ¿por qué parece que se cuelga?
No está colgado.
Está en este estado:

Waiting for new emails...
👉 Eso significa:
No hay correos nuevos
Está escuchando
No termina nunca (por diseño)
⚙️ 2. Si quieres que revise cada cierto tiempo (modo polling)
Ahí sí debes cambiar la estrategia:
✅ Opción A (RECOMENDADA): usar CRON + Email Read
En vez de usar Email Trigger, haces esto:
🔁 Flujo correcto:

Cron → Email Read IMAP → Code (tu lógica)
🕒 Nodo CRON ejemplo
Mode: Every X minutes
Ejemplo:
cada 1 minuto
cada 5 minutos
📥 Nodo: Email Read (IMAP)
Configuración clave:
Operation: Get All
Mailbox: INBOX
Options:
✅ Only Unread
✅ Download Attachments
Limit: 10 o 20
👉 Esto evita que procese miles de correos
🔥 Ventajas de este enfoque
✔ Control total del tiempo
✔ No se queda “colgado”
✔ Puedes re-ejecutar manualmente
✔ Ideal para pipelines (como el tuyo de facturas)
⚡ 3. ¿Cuándo usar Email Trigger entonces?
Usa Email Trigger solo si quieres:
Procesamiento en tiempo real
Automatización continua tipo servidor
Pero:
❌ No sirve para debug cómodo
❌ No sirve para ejecuciones manuales
❌ Parece “colgado”
